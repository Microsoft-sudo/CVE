# Mass Assignment Vulnerability in AsgardCMS User Module Allows Privilege Escalation

**ID:** vuln-0001
**Severity:** CRITICAL
**Found:** 2026-03-25 05:53:05 UTC
**Target:** /workspace/Platform/Modules/User
**Endpoint:** /user/users (POST), /user/users/{id}/edit (PUT)
**Method:** POST, PUT
**CWE:** CWE-915
**CVSS:** 9.9

## Description

A critical mass assignment vulnerability exists in the AsgardCMS User module that allows attackers to manipulate user permissions and roles during user creation and updates. The vulnerability stems from incomplete request validation combined with dynamic $fillable configuration that includes sensitive fields like 'permissions'.

The User model's $fillable array is dynamically loaded from configuration and includes the 'permissions' field, which controls user access rights. However, the CreateUserRequest and UpdateUserRequest validation classes only validate basic user fields (first_name, last_name, email, password) while allowing unvalidated fields like 'permissions' and 'roles' to be mass assigned.

Additionally, the mergeRequestWithPermissions method in API controllers directly merges user-provided permissions into the data array without proper authorization checks beyond middleware validation.

## Impact

Successful exploitation allows attackers to:
- Create new users with administrative privileges
- Escalate existing user privileges to administrative levels
- Assign arbitrary roles to users
- Manipulate permission sets for any user
- Potentially gain full control over the CMS administration panel

This vulnerability enables complete privilege escalation and can lead to full system compromise if exploited by an attacker with user creation or update permissions.

## Technical Analysis

The vulnerability exists due to three critical design flaws:

1. **Incomplete Request Validation**: CreateUserRequest and UpdateUserRequest classes only validate basic user fields but allow unvalidated sensitive fields like 'permissions' and 'roles' to pass through.

2. **Dynamic Fillable Configuration**: The User model constructor dynamically sets the $fillable property from configuration (config/asgard.user.config.fillable), which includes 'permissions'. This allows mass assignment of permission data.

3. **Permission Manipulation Method**: The mergeRequestWithPermissions method in API controllers directly merges request permissions into the data array without additional authorization checks.

The attack flow:
1. Attacker gains access to user creation/update endpoints (either through compromised credentials or other vulnerabilities)
2. Attacker sends crafted request with additional 'permissions' and 'roles' fields
3. Request passes validation (since these fields aren't validated)
4. Data is processed by SentinelUserRepository which mass assigns all fillable fields
5. User permissions/roles are updated with attacker-controlled values

## Remediation

1. **Update Request Validation**:
   - Modify CreateUserRequest and UpdateUserRequest to explicitly deny unvalidated fields
   - Add validation rules for 'permissions' and 'roles' fields to ensure they are properly sanitized
   - Use Laravel's validated() method instead of all() to get only validated data

2. **Remove Sensitive Fields from Fillable**:
   - Remove 'permissions' from the fillable array in User model configuration
   - Handle permissions separately with explicit authorization checks
   - Implement a dedicated method for permission updates with proper validation

3. **Strengthen Authorization**:
   - Add additional authorization checks in repository methods
   - Implement field-level authorization for sensitive operations
   - Validate role assignments against user's current permissions

4. **Implement Input Sanitization**:
   - Add server-side validation for permission structures
   - Validate role IDs against existing roles
   - Implement maximum privilege limits based on current user's permissions

Example fixed CreateUserRequest:
```php
public function rules()
{
    return [
        'first_name' => 'required',
        'last_name' => 'required',
        'email' => 'required|unique:users|email',
        'password' => 'required|min:3|confirmed',
        'permissions' => 'sometimes|array',  // Explicit validation
        'roles' => 'sometimes|array',        // Explicit validation
        'roles.*' => 'exists:roles,id'       // Validate role existence
    ];
```
