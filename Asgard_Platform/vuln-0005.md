# Stored XSS Vulnerability in AsgardCMS Translation Module via Unsanitized Translation Values

**ID:** vuln-0005
**Severity:** HIGH
**Found:** 2026-03-25 05:57:36 UTC
**Target:** /workspace/Platform/Modules/Translation
**Endpoint:** /api/translation/update
**Method:** POST
**CWE:** CWE-79
**CVSS:** 8.0

## Description

The AsgardCMS Translation module contains a critical stored cross-site scripting (XSS) vulnerability that allows authenticated users with translation editing permissions to inject malicious JavaScript code that will be executed when translations are rendered. The vulnerability exists because translation values are stored directly in the database without sanitization and rendered without proper output encoding in the admin interface.

The vulnerability affects:
1. Translation update API endpoint (/api/translation/update)
2. Translation CSV import functionality
3. Translation admin interface rendering

Attack vectors:
- Direct API calls with malicious payloads
- CSV file imports containing XSS payloads
- Inline editing in the admin interface

## Impact

An attacker with translation editing permissions can inject malicious JavaScript that will execute in the context of any administrator viewing the translation management interface. This can lead to session hijacking, account takeover, and complete compromise of the CMS administration panel. The XSS payloads persist across sessions and affect all administrators.

## Technical Analysis

The vulnerability exists in multiple components:

1. **Translation Storage**: The TranslationTranslation entity stores user-controlled 'value' field directly without sanitization. The $fillable array includes 'value' allowing mass assignment.

2. **API Update Endpoint**: The TranslationController@update method accepts raw 'value' parameter and saves it directly to the database via TranslationRepository::saveTranslationForLocaleAndKey().

3. **CSV Import**: The TranslationsImporter reads CSV values and stores them directly without sanitization.

4. **Rendering**: The admin interface (index.blade.php) renders translation values using `{{ }}` which does NOT escape HTML content in Laravel Blade templates. The translations are displayed as clickable editable fields using X-Editable library.

5. **JavaScript Rendering**: The translations are rendered in JavaScript via AJAX calls and displayed without proper encoding, allowing script execution.

## Proof of Concept

1. Authenticate as a user with translation editing permissions
2. Send a POST request to /api/translation/update with:
   - locale: en
   - key: test.key
   - value: <script>alert(document.domain)</script>

3. Navigate to the translation management interface (/admin/translation/translations)
4. The XSS payload will execute in the administrator's browser

## Remediation

1. Implement proper output encoding in Blade templates:
   - Replace `{{ $translation }}` with `{!! $translation !!}` ONLY when HTML is intended
   - Use `{{ e($translation) }}` or `{!! $translation !!}` with HTML Purifier

2. Add input validation and sanitization:
   - Implement HTML sanitization before storing translation values
   - Use Laravel's built-in validation with custom rules
   - Strip or encode dangerous HTML/JavaScript content

3. Secure CSV import:
   - Validate and sanitize CSV content before processing
   - Implement content filtering for imported translations

4. Use Content Security Policy (CSP):
   - Implement CSP headers to mitigate XSS impact
   - Restrict script execution to trusted sources

5. Implement proper authorization checks:
   - Ensure only authorized users can modify translations
   - Add role-based access controls for translation management
