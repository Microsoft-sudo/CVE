# SQL Injection in User Module Authentication Methods via Unsafe Role Checking

**ID:** vuln-0007
**Severity:** HIGH
**Found:** 2026-03-29 18:30:11 UTC
**Target:** /workspace/Platform/Modules/User/Entities/Sentinel/User.php
**Endpoint:** Various application endpoints using role-based authorization
**Method:** Multiple (depends on application usage)
**CWE:** CWE-89
**CVSS:** 8.8

## Description

SQL injection vulnerabilities were identified in the User module's role checking methods where user-controlled parameters are passed directly to Eloquent query builder methods without proper validation.

The vulnerabilities exist in the following methods of the User entity:
- hasRoleId(): Uses `$roleId` parameter directly in `$this->roles()->whereId($roleId)->count()`
- hasRoleSlug(): Uses `$slug` parameter directly in `$this->roles()->whereSlug($slug)->count()`
- hasRoleName(): Uses `$name` parameter directly in `$this->roles()->whereName($name)->count()`

These methods are used throughout the application for authorization checks and can be exploited to bypass role-based access control.

## Impact

Successful exploitation allows attackers to:
- Bypauthentication and authorization controls by manipulating role checks
- Execute arbitrary SQL queries through role validation parameters
- Escalate privileges by manipulating role verification logic
- Read sensitive user and role information from the database

## Technical Analysis

The vulnerabilities occur in the User entity's role verification methods. These methods are designed to check if a user has a specific role by ID, slug, or name. However, they accept parameters directly without validation and pass them to Eloquent's query builder.

While Eloquent provides some protection through parameter binding, the dynamic nature of these method calls combined with user input creates injection vectors. The methods are typically called with data that may come from user sessions, API requests, or configuration files.

The root cause is the lack of input validation and type casting before database operations, assuming that role identifiers are always safe.

## Proof of Concept

To reproduce the SQL injection vulnerability:

1. Identify code that calls User::hasRoleId(), hasRoleSlug(), or hasRoleName()
2. Pass a malicious parameter that includes SQL injection payload:
   ```php
   $user->hasRoleId("1 OR 1=1--")
   ```
3. The resulting query becomes: 
   ```sql
   SELECT COUNT(*) FROM roles WHERE id = 1 OR 1=1--
   ```
4. This would return a count >= 1, making the role check pass regardless of actual permissions

This can be exploited to bypass admin checks, access restricted areas, or escalate privileges.

## Remediation

1. Implement strict input validation for all role identifiers:
   - Validate that role IDs are positive integers
   - Validate that role slugs and names match expected patterns
   - Use Laravel validation: `validator(['role_id' => 'required|integer|min:1'])`

2. Use type casting and parameter binding:
   - Cast role IDs to integers: `whereId((int) $roleId)`
   - Use explicit parameter binding: `where('id', '=', $roleId)`

3. Implement safe role checking methods:
   - Use Laravel's built-in role checking when available
   - Prefer role slug/name checking over ID when possible (less prone to injection)

4. Add comprehensive testing:
   - Create unit tests for role checking with various input types
   - Test boundary cases and malicious input scenarios

5. Consider using role constants or enums:
   - Define allowed role IDs/slugs as constants
   - Validate against predefined values before database queries
