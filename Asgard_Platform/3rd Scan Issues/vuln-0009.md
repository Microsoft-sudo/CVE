# Critical Mass Assignment Vulnerability Allows Privilege Escalation via User Registration

**ID:** vuln-0009
**Severity:** CRITICAL
**Found:** 2026-03-29 18:45:59 UTC
**Target:** /workspace/Platform
**Endpoint:** /api/auth/register, /api/user/profile, and other user management endpoints
**Method:** POST, PATCH, PUT
**CWE:** CWE-915
**CVSS:** 10.0

## Description

A critical mass assignment vulnerability was identified in the Platform application's User module that allows attackers to escalate privileges through user registration or profile updates. The vulnerability exists because the 'permissions' field is included in the fillable array, making it susceptible to mass assignment attacks.

The User model configuration at `/workspace/Platform/config/asgard/user/config.php` includes 'permissions' in the fillable fields array (line 61). This allows user-controlled permissions to be mass assigned during registration or update operations without proper authorization checks.

This vulnerability enables attackers to grant themselves administrative privileges by including permission data in registration or profile update requests.

## Impact

Successful exploitation allows an attacker to:
- Grant themselves administrative privileges during user registration
- Escalate privileges through profile updates
- Gain full administrative access to the application
- Bypass all access controls and authorization mechanisms
- Potentially compromise the entire application and all user data

The vulnerability affects both new user registration and existing user profile updates, making it accessible to both unauthenticated and authenticated attackers.

## Technical Analysis

The vulnerability stems from improper mass assignment protection in the User model configuration:

```php
'fillable' => [
    'email',
    'password',
    'permissions',  // VULNERABLE - should never be mass assignable
    'first_name',
    'last_name',
],
```

The 'permissions' field should be guarded and only modifiable through proper authorization checks. Instead, it's exposed to mass assignment, allowing attackers to manipulate their own permissions.

**Root Cause Analysis:**
1. The User model inherits from Cartalyst Sentinel's EloquentUser
2. The fillable array is populated from configuration (line 36 in User.php)
3. No server-side validation prevents permission manipulation
4. No authorization checks validate permission changes

**Attack Vector:**
- Registration endpoint: POST request with malicious permissions payload
- Profile update endpoint: PUT/PATCH request with permission escalation
- Any endpoint that accepts user data for mass assignment

## Proof of Concept

To reproduce the mass assignment vulnerability:

1. **Registration Attack:**
```bash
curl -X POST http://target-domain.com/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "email": "attacker@example.com",
    "password": "password123",
    "permissions": {"admin": true, "superuser": true}
  }'
```

2. **Profile Update Attack:**
```bash
curl -X PATCH http://target-domain.com/api/user/profile \
  -H "Authorization: Bearer [user-token]" \
  -H "Content-Type: application/json" \
  -d '{
    "first_name": "Attacker",
    "permissions": {"admin": true}
  }'
```

## Remediation

1. **Remove permissions from fillable array:**
```php
'fillable' => [
    'email',
    'password',
    'first_name',
    'last_name',
    // REMOVED: 'permissions' - should never be mass assignable
],
```

2. **Implement proper permission management:**
```php
// Only allow permission changes through authorized methods
public function setPermissions(array $permissions)
{
    if (!auth()->user()->hasPermission('user.manage_permissions')) {
        abort(403, 'Unauthorized permission modification');
    }
    
    $this->attributes['permissions'] = json_encode($permissions);
}
```

3. **Use guarded instead of fillable for sensitive fields:**
```php
protected $guarded = ['permissions', 'roles', 'is_admin'];
```

4. **Implement request validation for user operations:**
```php
$request->validate([
    'email' => 'required|email',
    'first_name' => 'required|string',
    'last_name' => 'required|string',
    // Do not include permissions in validation rules
]);
```

5. **Use form requests with authorization checks:**
```php
class UpdateUserRequest extends FormRequest
{
    public function authorize()
    {
        return $this->user()->can('update', $this->user);
    }
    
    public function rules()
    {
        return [
            'email' => 'required|email',
            // Exclude permissions from rules
        ];
    }
}
```

6. **Conduct security review of all fillable arrays**
7. **Implement automated security testing for mass assignment
