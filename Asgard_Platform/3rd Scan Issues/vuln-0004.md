# SQL Injection Vulnerability in Search Functionality Across Multiple Modules

**ID:** vuln-0004
**Severity:** HIGH
**Found:** 2026-03-29 18:27:33 UTC
**Target:** /workspace/Platform
**Endpoint:** Multiple endpoints including /api/*/search routes and admin panel search functionality
**Method:** GET, POST
**CWE:** CWE-89
**CVSS:** 8.8

## Description

A critical SQL injection vulnerability was identified in the search functionality across multiple modules of the AsgardCMS Platform. The vulnerability exists due to improper handling of user input in LIKE queries where user-controlled data is directly interpolated into SQL statements without proper parameterization or escaping.

The vulnerability affects:
- Page module: Search functionality in EloquentPageRepository.php
- User module: User and role search in Sentinel repositories  
- Media module: File search functionality
- Translation module: Key search functionality

In each case, the user-supplied search term `$term` is directly interpolated into LIKE queries using `"%{$term}%"` pattern, allowing an attacker to inject malicious SQL code.

## Impact

Successful exploitation allows an attacker to:
- Execute arbitrary SQL queries on the database
- Extract sensitive information including user credentials, personal data, and system configurations
- Modify or delete database records
- Potentially achieve remote code execution through database functions

The vulnerability affects all users with access to search functionality, which typically includes authenticated users and potentially public endpoints depending on module configuration.

## Technical Analysis

The vulnerability stems from improper use of Laravel's Eloquent ORM. While Eloquent provides parameterized query capabilities, the code uses direct string interpolation for LIKE queries:

```php
$query->where('title', 'LIKE', "%{$term}%");
```

Instead of using proper parameter binding:
```php
$query->where('title', 'LIKE', '%' . $term . '%');
```

The `$term` variable comes directly from user input via `$request->get('search')` without any sanitization. An attacker can inject SQL syntax that gets executed directly by the database.

Root cause: Lack of input validation and improper use of Eloquent's parameter binding capabilities for LIKE queries.

## Proof of Concept

1. Navigate to any search functionality in the application (pages, users, media, etc.)
2. Enter a SQL injection payload as the search term, for example:
   - `' OR '1'='1' -- ` (basic authentication bypass)
   - `test%' UNION SELECT username,password FROM users -- ` (data extraction)
   - `test%'; DROP TABLE users; -- ` (destructive attack)

3. Observe that the application executes the malicious SQL query
4. For blind SQL injection, use timing-based payloads:
   - `test%' AND SLEEP(5) -- ` (MySQL)
   - `test%' AND pg_sleep(5) -- ` (PostgreSQL)

The vulnerability is exploitable through both GET and POST requests depending on the endpoint implementation.

## Code Analysis

**Location 1:** `Modules/Page/Repositories/Eloquent/EloquentPageRepository.php` (lines 148-151)
  SQL Injection vulnerability in page search functionality
  ```
              $pages->whereHas('translations', function ($query) use ($term) {
                $query->where('title', 'LIKE', "%{$term}%");
                $query->orWhere('slug', 'LIKE', "%{$term}%");
            })
  ```

  **Suggested Fix:**
```diff
-             $pages->whereHas('translations', function ($query) use ($term) {
-                 $query->where('title', 'LIKE', "%{$term}%");
-                 $query->orWhere('slug', 'LIKE', "%{$term}%");
-             })
+             $pages->whereHas('translations', function ($query) use ($term) {
```

## Remediation

1. Use proper parameter binding for LIKE queries:
```php
// Correct approach
$query->where('title', 'LIKE', '%' . $term . '%');

// Or using named parameters
$query->where('title', 'LIKE', DB::raw('?'), ['%' . $term . '%']);
```

2. Implement input validation and sanitization:
```php
$term = $request->get('search');
$validatedTerm = preg_replace('/[^a-zA-Z0-9\s\-_]/', '', $term); // Remove special chars
```

3. Use Laravel's built-in validation:
```php
$validated = $request->validate([
    'search' => 'required|string|max:255|regex:/^[a-zA-Z0-9\s\-_]+$/'
]);
```

4. Consider using prepared statements for complex queries
5. Implement rate limiting on search endpoints to prevent automated attacks
6. Add WAF rules to detect and block SQL injection attempts
