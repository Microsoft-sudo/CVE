# Stored XSS Vulnerability in AsgardCMS Page Module via Unsanitized Content Rendering

**ID:** vuln-0007
**Severity:** CRITICAL
**Found:** 2026-03-25 05:59:36 UTC
**Target:** /workspace/Platform/Modules/Page
**Endpoint:** /admin/page/pages (creation) and /{uri} (viewing)
**Method:** POST (creation) and GET (viewing)
**CWE:** CWE-79
**CVSS:** 9.0

## Description

A critical stored cross-site scripting (XSS) vulnerability exists in the AsgardCMS Page module where page body content is rendered without proper sanitization. The vulnerability allows attackers to inject and execute arbitrary JavaScript code in the context of any user viewing the affected page.

The vulnerability occurs in the content rendering flow where the page body content is passed through the `@editor` Blade directive and ultimately rendered using the unsafe `{!! !!}` syntax, which outputs raw HTML without encoding. This allows malicious scripts to be stored in page content and executed when the page is viewed by other users.

Additionally, the template loading mechanism uses user-controlled input without proper validation, creating potential for template injection attacks.

## Impact

Successful exploitation allows attackers to:
- Execute arbitrary JavaScript in the context of any user viewing the page
- Steal session cookies and authentication tokens
- Perform actions on behalf of authenticated users
- Deface pages or redirect users to malicious sites
- Potentially chain with other vulnerabilities for full account compromise

The vulnerability affects all pages created through the AsgardCMS admin interface and impacts all users who view the malicious pages.

## Technical Analysis

The vulnerability exists in the content rendering pipeline:

1. **Page Creation**: When a page is created or updated, the body content is stored in the database without sanitization
2. **Content Rendering**: The `PublicController::uri()` method retrieves the page and calls `view($page->template)` with user-controlled template
3. **Template Loading**: The template system uses `view()->exists()` to check template existence but doesn't sanitize the path
4. **Unsafe Output**: In `core::components/textarea-wrapper.blade.php`, line 2: `{!! $content !!}` renders raw HTML without encoding
5. **Editor Integration**: The `@editor` directive in page admin forms provides a rich text editor that can be used to inject scripts

The core issue is the use of `{!! !!}` (raw output) instead of `{{ }}` (escaped output) for user-controlled content. This violates the fundamental security principle of output encoding.

## Proof of Concept

To reproduce the stored XSS vulnerability:

1. Access the AsgardCMS admin interface
2. Navigate to Page module and create a new page
3. In the page body content, insert a JavaScript payload:
   ```html
   <script>alert(document.domain)</script>
   ```
4. Save the page and view it as a regular user
5. The JavaScript alert will execute, demonstrating XSS

For template injection:
1. Create a page with a malicious template path
2. Set template field to a path traversal payload or custom template
3. The system will attempt to load the specified template


## Code Analysis

**Location 1:** `Modules/Core/Resources/views/components/textarea-wrapper.blade.php` (line 2)
  Unsafe HTML output without encoding
  ```
  {!! $content !!}
  ```

  **Suggested Fix:**
```diff
- {!! $content !!}
+ {{ $content }}
```

**Location 2:** `Modules/Page/Http/Controllers/PublicController.php` (line 46)
  User-controlled template loading without validation
  ```
          $template = $this->getTemplateForPage($page);
  ```

**Location 3:** `Modules/Page/Http/Controllers/PublicController.php` (lines 92-94)
  Template path validation only checks existence, not safety
  ```
      private function getTemplateForPage($page)
    {
        return (view()->exists($page->template)) ? $page->template : 'default';
    }
  ```

## Remediation

1. **Sanitize Output**: Replace `{!! $content !!}` with `{{ $content }}` in all template files to enable automatic HTML encoding
2. **Content Sanitization**: Implement server-side HTML sanitization using libraries like HTMLPurifier before storing page content
3. **Template Validation**: Add strict validation for template paths to prevent path traversal and arbitrary file inclusion
4. **Content Security Policy**: Implement CSP headers to mitigate the impact of successful XSS attacks
5. **Input Validation**: Validate and sanitize all user input, especially HTML content, before storage

Specific code changes needed:
- In `/workspace/Platform/Modules/Core/Resources/views/components/textarea-wrapper.blade.php`: Change `{!! $content !!}` to `{{ $content }}`
- In page repository: Add HTML sanitization before storing page body content
- In template loading: Add path normalization and validation
